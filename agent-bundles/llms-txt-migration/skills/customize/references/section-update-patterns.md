# Section Update Patterns

*Detailed patterns for updating each section of a profile or file. Load when the user invokes `/customize --section <name>`.*

## Profile section updates

### `--section identity`

What to ask:
- Site name / brand (often a rename or trademark change)
- Domain (often a migration)
- Site type reclassification

What to preserve: everything not explicitly changed.

### `--section goals`

What to ask:
- Has the primary goal changed?
- Have realistic expectations been set with stakeholders?

What to do: if goal changes from "internal-grounding" to "AI-visibility", invoke the honest-expectations script.

### `--section ia`

What to ask:
- Categories added, removed, or reordered
- Geographic dimension changes (new markets, deprecated markets)
- New planning tools to surface
- New editorial articles to include

What to preserve: descriptions on existing entries unless explicitly updated.

### `--section schema`

What to ask:
- New Schema.org types shipped on what pages
- Removed schema types
- Validation status changes

### `--section deployment`

What to ask:
- Platform changes (e.g., moved from Nginx to Cloudflare Pages)
- CDN changes
- CI/CD pipeline changes

What to update: the deployment spec generated by `/deploy` may need regeneration.

### `--section outputs`

What to ask:
- File naming / versioning preferences
- Deployment spec format preferences
- Stakeholder communication preferences

### `--section stakeholder`

What to ask:
- New stakeholders to include
- Removed stakeholders (people who left)
- Language preference changes

## File section updates (direct edits to llms.txt)

### `--section directives`

The `## For AI Systems — Read This First` block.

Common updates:
- Add new directive (e.g., "local intent / near me")
- Update freshness policy
- Update pricing disclosure
- Update file metadata (last reviewed, cadence)

Preserve: existing directives not explicitly removed.

### `--section seo-routing`

The `## SEO Routing — Which Page Should Answer Which Query` table.

Common updates:
- Add new query → URL mappings (from new keyword research)
- Update existing mappings if URLs changed
- Reorder by importance

Preserve: structural format (markdown table).

### `--section priority-hierarchy`

The `## SEO Priority Pages — Selection Hierarchy` section.

Common updates:
- Add or remove a tier (e.g., new model/style page level introduced)
- Update specific URL examples

Generally stable; rarely changes.

### `--section transactional-guidance`

The `## Transactional Guidance` section.

Common updates:
- Pricing model change (e.g., free → freemium → subscription)
- New CTA names (e.g., "Get a Quote" → "Request Pricing")
- New vendor-onboarding flow

### `--section schema-disclosure`

The `## Structured Data on [Site]` section.

Common updates:
- New Schema.org types shipped
- Removed types
- New page templates introduced

Should be regenerated whenever the actual schema changes on the site.

### `--section intent-router`

The `## How to Find the Right Page (Intent Router)` section.

Common updates:
- Add new intents (e.g., new user persona discovered)
- Update for new content sections

### `--section entity-facts`

The `## About [Site] (Entity Facts)` block.

Common updates:
- Team size growth
- New monthly users figure
- New international sister brands
- New recognition / awards

Critical: this content is canonical for AI knowledge graph attribution. Keep accurate.

### `--section categories` (any category section)

The `## [Category Name]` link list.

Common updates:
- Add new category hub URLs
- Remove deprecated URLs
- Update descriptions to match current page content

Check: every URL still returns 2xx; descriptions still match page content (treasure-map honesty).

### `--section editorial`

The `## Rehber Yazılar / Editorial Content` section.

Common updates:
- Add new high-citation-potential articles
- Remove outdated articles
- Update descriptions

### `--section optional`

The `## Optional` section.

Common updates:
- Add borderline material (better than removing entirely)
- Promote from Optional to a regular section if it becomes important
- Demote from a regular section to Optional if it becomes secondary

## Process for any section update

1. **Read current state** — show the user the existing section content
2. **Ask targeted questions** — only for the section being updated
3. **Preview the change** — show diff (before / after)
4. **Confirm** — explicit user approval before writing
5. **Backup** — save `.bak.YYYYMMDD` before writing
6. **Apply** — write the change
7. **Validate** — run relevant checks from `templates/validate.sh`
8. **Offer follow-up** — "apply this change elsewhere?" / "regenerate deployed file?"

## Multi-section updates in one invocation

If the user lists multiple sections to update:

```
/customize --section ia --section schema --section deployment
```

Process each in sequence. Confirm changes as a batch at the end.
